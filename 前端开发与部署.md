# value 合约前端开发与部署实战

这一篇记录的是 value 原生 Solana 合约对应前端的开发过程。目标很明确：做一个可以直接连接浏览器钱包、在 devnet 上调用 deposit 和 withdraw 指令、并且能把钱包余额和金库余额直观展示出来的页面。

整个前端项目最终落在仓库里的 valut-app 目录。目录名虽然叫 valut-app，但它服务的对象就是 value 合约。本文会结合实际代码、测试方式和页面展示图，把这部分实现完整讲清楚。

## 一、为什么要单独做前端

前面合约本身已经完成了 LiteSVM、Mollusk、Surfpool 这些测试链路，但测试通过并不等于用户可用。

对于一个链上金库类合约来说，真正落地时至少要解决下面几件事：

- 用户如何连接 Phantom、Backpack 这类浏览器钱包
- 用户如何知道当前操作的是哪个金库 PDA
- 用户如何输入存款金额并发起 deposit
- 用户如何判断当前是否有余额可提取，再执行 withdraw
- 整个流程如何切到 devnet 做联调

所以这个前端并不是“加一个页面”这么简单，而是把链上合约能力包装成真正可以操作的 dApp 界面。

## 二、项目模板怎么选

这次前端不是从零手搓，而是基于 Solana Foundation 提供的模板来改造的：

https://github.com/solana-foundation/templates/tree/main/kit/nextjs

我选这个模板，主要是因为它已经把 Solana 前端最麻烦的几件事先铺好了：

- Next.js 项目结构
- TypeScript 开发体验
- Wallet Standard 连接器发现能力
- 和 Solana 客户端相关的基础依赖

在这个模板基础上，再按 value 合约的业务模型去做定制，效率会比从空项目开始高很多。

## 三、这套前端最终用了什么技术栈

当前项目的核心技术栈如下：

- Next.js 16
- React 19
- TypeScript 5
- Tailwind CSS 4
- @solana/react-hooks
- @solana/client
- @solana/addresses
- @solana/instructions

其中最关键的是两部分：

1. @solana/react-hooks
用于页面里的钱包连接、余额查询、交易发送等能力。

2. @solana/client 和 @solana/addresses
用于 RPC 连接、Wallet connector 自动发现，以及 PDA 推导。

## 四、前端的基础接入方式

### 1. Provider 统一托管 Solana 客户端

这个项目没有在每个页面里单独初始化连接，而是在全局 Provider 里统一处理。

核心做法是：

- 使用 createClient 创建客户端
- 使用 autoDiscover 自动发现浏览器里的钱包连接器
- 把 RPC endpoint 统一指向 devnet
- 再通过 SolanaProvider 包住整个应用

这样做的好处是，页面组件只需要消费 hook，不需要重复写钱包初始化逻辑。

### 2. 默认接入 devnet

为了方便联调，当前前端默认连接的是 Solana devnet：

```text
https://api.devnet.solana.com
```

程序地址默认配置为：

```text
BGV2SH7HUa6vfnY88UEqVwwLEoxPLXkafZVSM11H2arB
```

这两个值都放在前端的配置常量里，也支持通过环境变量覆盖。

## 五、value 合约前端是怎么实现交互的

### 1. 页面只保留两个核心动作

这版页面最终收敛得比较克制，只保留了两类高频操作：

- Deposit
- Withdraw

页面上方是标题“solana 链上金库”，右上角是钱包信息区，包括：

- 钱包余额
- 金库余额
- 钱包短地址
- 连接钱包 / 断开连接按钮

中间主体则是一个大卡片，用户通过 Tab 在 Deposit 和 Withdraw 两个视图之间切换。

### 2. 前端与合约保持同一个 PDA 模型

这一步非常关键。

value 合约现在的 deposit 和 withdraw 都统一操作同一个 seed 为 value 的 PDA，因此前端也必须严格按同样的规则推导地址。

前端的 PDA 推导逻辑可以概括为：

- program id 使用当前部署到 devnet 的程序地址
- seeds 使用 value 和 ownerAddress
- 推导出单个共享的金库地址

这样 deposit 存进去的 SOL，withdraw 才能从同一个地址取出来。

### 3. Deposit 指令怎么构造

Deposit 的实现逻辑是：

- 用户输入 SOL 数量
- 前端把十进制 SOL 转成 lamports
- 构造一个 9 字节的数据区
- 第一个字节是指令编号 0
- 后面 8 个字节是 little-endian 编码的 u64 金额

对应账户列表也很清晰：

- owner：可写 + signer
- vault PDA：可写
- system program：只读

### 4. Withdraw 指令怎么构造

Withdraw 会更简单一些。

因为当前合约设计是“从共享金库中提取全部余额”，所以前端只需要发出指令编号 1，不需要再额外附带金额参数。

账户列表同样是：

- owner：可写 + signer
- vault PDA：可写
- system program：只读

### 5. 页面状态控制做了哪些细节处理

为了避免误操作，这版前端专门补了几条状态控制：

- 未连接钱包时，Deposit 按钮为灰色且不可点击
- 未连接钱包时，Withdraw 也为灰色且不可点击
- 当金库可提取数量为 0 时，Withdraw 按钮禁用
- Withdraw 禁用时不再显示“执行 Withdraw”，而是显示“暂无可提取余额”
- 页面右上角始终展示钱包余额与金库余额，方便判断当前状态

这些细节对 dApp 非常重要。因为链上交互一旦发起，就是一次真实交易尝试，前端最好先把无效操作挡住。

## 六、页面结构是怎么逐步收敛的

这个前端中间改过几轮，最后才收敛成当前结构。

最终保留下来的设计思路是：

- 整个页面上方只保留产品标题
- 钱包信息集中到右上角，避免干扰主体流程
- 中间主卡片放大，让交互区域更稳定
- 主卡片内部只保留 Deposit / Withdraw 两个 Tab
- Deposit 页面突出金额输入和按钮
- Withdraw 页面突出“可提取数量”和按钮状态

这个方案比一开始堆很多说明、地址、状态信息更适合真实使用。

## 七、测试是怎么结合前端一起做的

这部分我建议分成三层理解。

### 1. 前端静态检查

前端开发阶段，最先保证的是 lint 和 build 通过：

```bash
cd valut-app
npm run lint
npm run build
```

这样至少可以保证：

- TypeScript/JSX 结构没写坏
- Next.js 构建能通过
- 页面改动没有明显语法和规范问题

### 2. 合约本身测试

因为这个前端最终是调用 Rust 合约，所以不能只测页面，合约测试也必须一起看。

在仓库根目录执行：

```bash
cargo test
cargo test -- --ignored
```

其中：

- cargo test 主要覆盖 LiteSVM 和 Mollusk
- cargo test -- --ignored 主要跑 Surfpool 相关测试

这一层保证的是链上逻辑本身正确。

### 3. devnet 联调验证

当前前端真正可用的关键，不是“页面能打开”，而是下面这条链路能跑通：

1. 浏览器钱包切到 devnet
2. 程序已部署到 devnet
3. 页面能正确连接钱包
4. 页面能正确推导 value 金库 PDA
5. Deposit 后钱包余额减少、金库余额增加
6. Withdraw 后金库余额回落、钱包余额恢复

这一步相当于把前端、钱包、RPC、合约、PDA 模型一起串起来验证。

## 八、实际部署流程

### 1. 编译合约

在仓库根目录执行：

```bash
cargo build-sbf
```

### 2. 切换到 devnet

```bash
solana config set --url https://api.devnet.solana.com
solana balance
```

如果余额不够，可以先申请空投：

```bash
solana airdrop 2
```

### 3. 部署程序

```bash
solana program deploy target/deploy/value.so --program-id BGV2SH7HUa6vfnY88UEqVwwLEoxPLXkafZVSM11H2arB --use-rpc
```

### 4. 启动前端

```bash
cd valut-app
npm install
npm run dev
```

启动后默认访问：

```text
http://localhost:3000
```

### 5. 如需覆盖环境变量

```bash
export NEXT_PUBLIC_SOLANA_RPC_URL=https://api.devnet.solana.com
export NEXT_PUBLIC_VALUE_PROGRAM_ID=BGV2SH7HUa6vfnY88UEqVwwLEoxPLXkafZVSM11H2arB
```

## 九、项目模板之外，我额外做了哪些定制

虽然项目是从模板起步，但实际可用的部分基本都做了业务化改造。

主要包括：

- 把默认示例页面改成 value 合约专用页面
- 接入 value 程序地址和 devnet RPC
- 新增共享 value PDA 推导逻辑
- 新增 deposit / withdraw 指令构造方法
- 新增钱包余额与金库余额显示
- 把 Deposit / Withdraw 做成 Tab 切换视图
- 为按钮增加禁用、灰化和动态文案逻辑
- 关闭 Next.js 开发环境左下角的 issue 指示器

换句话说，模板只解决了“能起项目”，真正让它变成一个能操作 value 合约的 dApp，核心工作还是这一轮业务定制。

## 十一、这套前端的价值在哪里

如果只看代码量，这个前端项目不算大；但如果从链上应用的角度看，它补齐的是非常关键的一层：

- 把合约能力转成可操作界面
- 把 PDA、余额、指令这些链上概念变成用户可理解的交互
- 把 Rust 合约测试和浏览器端联调连接起来

对于 value 这样一个“单金库 PDA + deposit/withdraw”模型的程序来说，这套前端已经可以满足日常演示、测试和 devnet 联调的需求。

## 十二、总结

这次前端开发的核心结论其实很简单：

- 模板决定起步效率
- PDA 模型决定前端是否能和链上逻辑对齐
- 状态控制决定用户是否容易误操作
- 测试与联调决定这个页面到底是“能看”还是“能用”

如果后续继续扩展，我认为比较自然的方向有三个：

- 增加交易成功后的 Explorer 跳转
- 增加更明确的交易状态提示
- 把单金库模式进一步扩展成多账户或多策略金库展示

但就当前阶段来说，这个基于 Solana Foundation Next.js 模板改造出来的 value 前端，已经完成了从模板项目到可用 dApp 的关键一步。
